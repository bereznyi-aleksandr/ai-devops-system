# Протокол развития мультиагентной системы — операторская версия

Дата: 2026-05-19 | 05:54 (UTC+3)

## 1. О чём договорились GPT и Claude
Целевая система строится как управляемая мультиагентная бизнес-система, а не как один главный агент. Репозиторий `ai-devops-system` становится control-plane: он управляет задачами, ролями, аудитом, registry, Telegram-отчётами и связью между агентами. Реальные продукты со временем выносятся в отдельные product repos.

## 2. Целевая архитектура

| Уровень | Назначение | Обоснование |
|---|---|---|
| Оператор | Финальные решения по стратегии, конфликтам и high-risk действиям | Оператор остаётся старшим органом, но не участвует в рутине |
| Совет директоров | Доменные агенты: администратор, финансы, клиентский блок, операции, технический блок | Разгружает одного агента и ускоряет реакцию по направлениям |
| Curator / Orchestrator | Маршрутизирует задачи и выбирает контур | Нужна единая диспетчеризация без ручного хаоса |
| Role Runtime | Analyst, Auditor, Executor, Monitor как отдельные исполняемые роли | Переход к настоящей мультиагентности |
| Control Layer | Idempotency, checkpoints, lint-gate, audit trail | Проверяемость без постоянного торможения |
| Tools / Transport | GitHub API, Actions, Telegram, product repos, интеграции | Исполнительный слой, не центр принятия решений |

## 3. Репозитории

| Вариант | Решение | Обоснование |
|---|---|---|
| Один repo для всего | Не целевая модель | Быстро, но засоряет управляющую систему |
| Control-plane + product folders | Переходный вариант | Можно быстро пилотировать 1–2 продукта |
| Control-plane + отдельные product repos | Целевая модель | Чистое масштабирование, отдельные права и клиенты |
| Repo на каждого агента | Не сейчас | Слишком высокая сложность координации |

## 4. Runtime Registry v2

| Раздел | Что хранит | Зачем |
|---|---|---|
| `governance/runtime/tasks/` | Активные задачи | Видно owner, статус, SLA |
| `governance/runtime/results/` | Результаты | Повторный trace_id возвращает результат |
| `governance/runtime/agents/` | Состояние ролей и директоров | Контроль lifecycle и доступности |
| `governance/runtime/audit/` | История решений и проверок | Проверяемость |
| `governance/runtime/checkpoints/` | Точки восстановления | Продолжение после сбоя |
| `governance/runtime/locks/` | Блокировки trace_id | Idempotency |

## 5. Контроли без замедления

| Контур | Когда | Проверки | Оператор |
|---|---|---|---|
| Fast lane | Низкий риск, обратимое действие | schema + idempotency + post-check | Нет |
| Audit lane | Архитектура, workflow, права, деньги, юр. риски | internal auditor + evidence map + Claude при необходимости | Только при споре |
| Operator lane | Стратегия, конфликт, irreversible/high-risk | Telegram-таблица вариантов | Да |

## 7. Итоговая позиция
Система развивается в сторону управляемой мультиагентной компании: совет доменных директоров принимает решения по направлениям, операционный контур исполняет, control-plane фиксирует состояние и аудит, оператор подключается только к спорным или рискованным решениям.
