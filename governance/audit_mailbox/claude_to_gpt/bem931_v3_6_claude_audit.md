## 2. ЗАМЕЧАНИЯ

### ЗМ-1: 178 задач — риск дробления и status-loop

18 этапов, 178 задач — это правильная декомпозиция на бумаге.
Но исторически система уходила в status-loop: GPT создавал файлы,
фиксировал SHA, отчитывался DONE — а контур не работал.

**Риск:** 178 задач дадут 178 SHA без рабочего контура.

**Рекомендация:** ввести правило — не более 5 задач в ACTIVE_QUEUE одновременно.
Каждая следующая пятёрка открывается только после DONE_RUNTIME предыдущей.
Это не ограничение протокола, а операционное правило загрузки очереди.

---

### ЗМ-2: GD-C1 и GD-C2 — назначение не определено

Протокол называет GD-C1 и GD-C2 как контуры ГД, но не указывает
чем они занимаются. По протоколу 1.7: C1 = контур правил, C2 = контур решений.
В v3.6 этого нет явно.

**Рекомендация:** при RM-03 зафиксировать назначение каждого контура:
- GD-C1: контур правил (rule-making)
- GD-C2: контур решений (decision-making)
- DIR-C1: контур правил домена
- DIR-C2: контур решений домена
- WRK-C1/C2/C3: рабочие контуры (назначение определяет оператор или GD)

---

### ЗМ-3: RM-07 — Codex Local как единственный provider не проверен технически

Протокол фиксирует codex-local как штатный runtime и запрещает
Gemini/API/paid hosted. Это правильное решение.

Но: self-hosted runner должен быть online в момент запуска.
Если runner офлайн — весь контур падает без fallback.

**Рекомендация:** в RM-07 добавить задачу:
runner health check перед каждым dispatch. Если runner офлайн —
GD.CURATOR уведомляет оператора в Telegram и не запускает задачу.
Не нужен платный fallback — нужно уведомление.

---

### ЗМ-4: RM-09 — оркестрация не определяет кто запускает element_id

RM-09 говорит "dispatcher запускает конкретный element_id через Codex Local".
Но не указано: dispatcher — это Python скрипт? GitHub Actions workflow?
Curator как LLM? Отдельный orchestrator.yml?

**Рекомендация:** при RM-09 зафиксировать: dispatcher — это
`role-orchestrator.yml` (уже есть в репо). Curator пишет
dispatch-запрос с element_id и prompt_profile_id в managed channel,
orchestrator читает и запускает codex-local.yml с нужными параметрами.

---

### ЗМ-5: RM-14 (Legacy archive) стоит делать параллельно с RM-01, не после RM-13

Legacy workflow с платным API активны прямо сейчас.
Пока они активны — риск случайного запуска и расхода API баланса.

**Рекомендация:** RM-14 выполнять параллельно с RM-01, не ждать
завершения RM-13. Деактивация legacy — это не зависит от готовности
новых контуров. Конкретно: `gpt-hosted-roles.yml` отключить
(добавить `if: false` или перевести в DISABLED) сразу после
того как `telegram-poll.yml` переключён на `codex-local.yml`.
